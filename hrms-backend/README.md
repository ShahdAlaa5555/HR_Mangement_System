## 1. System Overview

The HRMS backend is a RESTful API server serving four fully integrated HR modules.
All implementation code was generated by Claude 3.5 Sonnet using structured
architectural prompts, with iterative refinement applied through targeted
debugging cycles.

| Module | Purpose | Key Entities |
|---|---|---|
| Employee Profile | Central employee data store | Employee, Department, Position, WorkLocation |
| Time & Attendance | Clock-in/out, shifts, corrections | AttendanceRecord, Shift, OvertimeRequest |
| Leave Management | Request lifecycle, balances, approvals | LeaveRequest, LeaveBalance, LeaveApproval |
| Payroll | Salary calculation, payslips | PayrollRun, PayrollEntry, Payslip |

All four modules share:
- A single SQL Server database (46 tables including junction and audit tables)
- Centralised JWT authentication
- Unified error response format
- A notification system for all user-facing events
- Audit logging for compliance

## 2. Architecture and Design Rationale

### 2.1 Layered Architecture

```
HTTP Request
    ↓
[Router] — URL mapping, middleware application, role guards
    ↓
[Middleware: auth.js] — JWT verification, req.user attachment
    ↓
[Middleware: validate.js] — Joi schema validation
    ↓
[Controller] — request parsing, response formatting
    ↓
[Service] — ALL business logic lives here
    ↓
[Prisma Client] — type-safe database access
    ↓
[SQL Server Database]
```

This separation was a core architectural requirement of the thesis:
- **Testability**: services are unit-tested without an HTTP server
- **Maintainability**: each layer has a single responsibility
- **AI generation quality**: the clear layer boundaries were provided in PT-1
  to guide the LLM toward a consistent structure across all four modules

### 2.2 Why Prisma

Prisma was selected because it provides compile-time type checking of database
interactions, meaning AI-generated data access code is validated against the
schema automatically. This reduces the risk of silent mismatches between
application logic and underlying tables — a category of defect that would
otherwise only appear at runtime.

### 2.3 Why a Single Generation Pass

The entire backend was generated in a single PT-1 invocation rather than
module by module. This decision reflects the tight cross-module dependencies
of the HRMS: generating modules sequentially tends to introduce naming
inconsistencies and broken ORM traversals at module boundaries. A single
invocation with the full schema and requirements present ensures all four
modules share a consistent internal representation from the first generation pass.

## 4. Database Integration

### 4.1 Schema

46 tables total: 40 core UML-specified entities plus junction and audit tables
generated by Prisma. All 40 UML entities conform fully to the architectural
specification, verified through UML-to-JSON structural validation.

### 4.2 Key Relationships

```
Employee ────────┬──→ Department         (ManyToOne)
                 ├──→ Position           (ManyToOne)
                 ├──→ WorkLocation       (ManyToOne)
                 ├──→ Employee           (Self-reference, SupervisorID)
                 ├──→ AttendanceRecord   (OneToMany)
                 ├──→ LeaveRequest       (OneToMany)
                 ├──→ LeaveBalance       (OneToMany per LeaveType)
                 ├──→ PayrollEntry       (OneToMany)
                 └──→ Payslip            (OneToMany)

LeaveRequest ────┬──→ LeaveType
                 ├──→ LeaveApproval      (OneToMany)
                 └──→ AttendanceRecord   (cross-module link via LeaveRequestID)

PayrollEntry ────┬──→ PayrollRun
                 ├──→ AttendanceSummary  (cross-module link)
                 └──→ PayrollEntryLine   (OneToMany)
```

### 4.3 Cross-Module Foreign Keys

| FK | From | To | Purpose |
|---|---|---|---|
| `LeaveRequestID` | `AttendanceRecord` | `LeaveRequest` | Approved leave overrides attendance status |
| `AttendanceSummaryID` | `PayrollEntry` | `AttendanceSummary` | Payroll reads attendance data |

## 5. Module APIs

### 5.1 Authentication

**Base URL**: `/api/v1/auth`

| Method | Route | Auth | Description |
|---|---|---|---|
| POST | `/login` | Public | Authenticate with email + password |
| POST | `/refresh` | Public | Exchange refresh token |
| GET | `/me` | Required | Get current user |
| POST | `/change-password` | Required | Change own password |

**Login Request:**
```json
POST /api/v1/auth/login
{
  "email": "sarah.johnson@university.edu",
  "password": "HRMSPass123!"
}
```

**Login Response:**
```json
{
  "success": true,
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIs...",
    "refreshToken": "eyJhbGciOiJIUzI1NiIs...",
    "tokenType": "Bearer",
    "expiresIn": "8h",
    "employee": {
      "id": 1,
      "code": "EMP001",
      "name": "Sarah Johnson",
      "email": "sarah.johnson@university.edu",
      "role": "HR",
      "department": "Computer Science",
      "position": "Associate Professor"
    }
  }
}
```

**Role Resolution Logic:**
The system derives roles from Position titles at login time:
- `"Admin"` / `"System"` in title → Role: `Admin`
- `"HR"` / `"Human Resource"` in title → Role: `HR`
- `"Payroll"` / `"Compensation"` in title → Role: `Payroll`
- `"Manager"` / `"Director"` / `"Head"` / `"Dean"` → Role: `Manager`
- Anything else → Role: `Employee`

| GET | `/requests/:id` | Self or Manager+ | Request details and approval trail |
| POST | `/requests/:id/approve` | Manager/HR | Approve, reject, or delegate |
| POST | `/requests/:id/cancel` | Self | Cancel pending request |
| GET | `/types` | Any | List leave types |
| GET | `/policies` | HR/Admin | List leave policies |
| GET | `/holidays` | Any | Holiday calendar |
| GET | `/analytics` | HR/Admin | Leave trends and statistics |

**Multi-Level Approval Workflow:**
```
Employee submits → Status: PendingManager
    ↓
Manager approves → checks ApprovalStep table
    ↓
    If HR step required → Status: PendingHR
        ↓
        HR approves → Status: Approved
            ↓
            [INTEGRATION] LeaveBalance.UsedDays += TotalDays
            [INTEGRATION] AttendanceRecord rows marked OnLeave
            [NOTIFICATION] Employee + Manager notified
    ↓
    If no HR step → Status: Approved immediately
    ↓
    If rejected at any step → Status: Rejected
        ↓
        [INTEGRATION] LeaveBalance.PendingDays -= TotalDays (returned)
```

**Validation Rules on Submission:**
1. `EndDate >= StartDate`
2. `TotalDays <= RemainingDays` — returns `INSUFFICIENT_BALANCE` if violated
3. `MinTenureMonths` satisfied per LeavePolicy
4. `NoticePeriodDays` satisfied — StartDate minus today >= policy requirement
5. No overlapping approved leave for same employee
6. `IsHalfDay` only allowed if `LeavePolicy.AllowHalfDay = true`

## 6. Cross-Module Integration

### Leave → Attendance
When a leave request reaches `Approved` status, `leave.service.js` calls
`attendance.service.js::markAttendanceAsOnLeave()`. For each calendar day
in the approved leave period, attendance records are created or updated with
`Status = OnLeave` and `LeaveRequestID` linked. This prevents the payroll
module from counting approved leave days as unpaid absences.

### Attendance → Payroll
When a payroll run is processed, for each employee the calculator calls
`attendance.service.js::generateAttendanceSummary()` returning PresentDays,
AbsentDays, LeaveDays, and TotalOvertimeHours. The payroll calculator uses
this to compute absence deductions and overtime pay.
`PayrollEntry.AttendanceSummaryID` links to the summary for traceability.

### Leave → Payroll (Paid vs Unpaid)
- `LeaveType.IsPaid = true` → no deduction applied
- `LeaveType.IsPaid = false` → `UnpaidLeaveDeduction = DailyRate × UnpaidLeaveDays`

## 7. Business Logic Implementation

### Egyptian Income Tax (2025)

```
Annual Taxable Income = (GrossMonthly × 12) - PersonalExemption (20,000 EGP)

Bracket 1:  0 – 15,000         →  0%
Bracket 2:  15,001 – 30,000    →  2.5%
Bracket 3:  30,001 – 45,000    → 10%
Bracket 4:  45,001 – 60,000    → 15%
Bracket 5:  60,001 – 200,000   → 20%
Bracket 6:  200,001 – 400,000  → 25%
Bracket 7:  400,001+           → 27.5%

Monthly Tax = AnnualTax / 12
```

Tax brackets are stored in the `TaxBracket` table, making them updatable
without code changes when tax laws change.

### Social Insurance (Law No. 148/2019)

```
SI Wage               = min(BaseSalary + Allowances, SI_CEILING)
Employee Contribution = SI_Wage × 7.25%
Employer Contribution = SI_Wage × 18.75%
Net Pay               = GrossIncome - Employee_SI - IncomeTax
```

Configuration is stored in the `SocialInsuranceConfig` table with effective
date versioning.

### Overtime (Egyptian Labor Law No. 12/2003, Article 130)

```
DailyRate    = BaseSalary / WorkingDaysInMonth
HourlyRate   = DailyRate / ShiftExpectedHours
OvertimePay  = HourlyRate × OvertimeHours × Multiplier

Weekday OT:       Multiplier = 1.35  (35% premium)
Holiday/Rest OT:  Multiplier = 2.00  (100% premium)
Maximum per day:  2 hours — enforced at request submission and calculation
```

### Absence Deduction

```
WorkingDaysInMonth      = calendar days - weekends - public holidays
DailyRate               = BaseSalary / WorkingDaysInMonth
AbsenceDeduction        = DailyRate × min(AbsentDays, MaxMonthlyDeductionDays)
MaxMonthlyDeductionDays = 5 (configurable per PayrollPolicy)
```

## 8. Security Design

| Measure | Implementation | Purpose |
|---|---|---|
| JWT Authentication | jsonwebtoken HS256, 8h access + refresh token | Stateless session management |
| Password Hashing | bcryptjs cost factor 12 | Prevents plaintext storage |
| Role-Based Access | `authorize(...roles)` middleware | Least-privilege enforcement |
| Rate Limiting | express-rate-limit, 10 login attempts per 15 min | Brute-force prevention |
| HTTP Security Headers | helmet | XSS, clickjacking, MIME sniffing prevention |
| CORS | Explicit ALLOWED_ORIGINS list | Unauthorised cross-origin prevention |
| Input Validation | Joi schemas on every endpoint | Injection and malformed data prevention |
| Parameterised Queries | Prisma built-in | SQL injection prevention |
| Audit Logging | EmployeeAuditLog table | Compliance and forensic capability |

Five roles are defined: Employee, Manager, HR, Payroll, and Administrator.
Authorisation rules are enforced through centralised middleware at the Router
Layer, ensuring access control logic cannot be accidentally omitted from a
newly generated route.

## 9. Error Handling Strategy

All errors return a consistent envelope:

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable description",
    "details": [
      { "field": "StartDate", "message": "StartDate cannot be in the past" }
    ]
  }
}
```

| HTTP Status | Code | Cause |
|---|---|---|
| 400 | `INSUFFICIENT_BALANCE` | Not enough leave days |
| 400 | `ALREADY_CHECKED_IN` | Duplicate check-in |
| 400 | `OVERTIME_LIMIT_EXCEEDED` | More than 2 hours OT requested |
| 401 | `INVALID_CREDENTIALS` | Wrong email or password |
| 401 | `TOKEN_EXPIRED` | JWT expired |
| 403 | `FORBIDDEN` | Insufficient role |
| 404 | `NOT_FOUND` | Record does not exist |
| 409 | `DUPLICATE_ENTRY` | Unique constraint violation |
| 422 | `VALIDATION_ERROR` | Joi schema failed |
| 500 | `INTERNAL_ERROR` | Unexpected server error |

Prisma error codes are automatically translated:
- `P2002` → 409 DUPLICATE_ENTRY
- `P2025` → 404 NOT_FOUND
- `P2003` → 400 FOREIGN_KEY_ERROR

## 10. Setup and Running

### Prerequisites
- Node.js >= 18.0.0
- SQL Server 2019+ or Azure SQL
- npm >= 9.0.0

### Steps

```bash
# 1. Install dependencies
cd hrms-backend
npm install

# 2. Configure environment
cp .env.example .env
# Set DATABASE_URL and JWT_SECRET in .env

# 3. Run the database schema
sqlcmd -S localhost -U sa -P YourPassword -i schema.sql

# 4. Generate Prisma client
npx prisma generate

# 5. Start development server
npm run dev

# 6. Verify
curl http://localhost:3000/health
# Expected: {"status":"ok","service":"University HRMS API"}
```

## 12. Design Assumptions

| # | Assumption | Justification |
|---|---|---|
| 1 | Egyptian work week is Sunday–Thursday | Standard in Egypt; `countEgyptianBusinessDays()` skips Friday and Saturday |
| 2 | Role derived from Position title string matching | Thesis simplification; production would use a dedicated UserRole table |
| 3 | Payslip DocumentURL is a file path placeholder | Full PDF generation is a production extension; out of thesis scope |
| 4 | Bank file is JSON structure, not a real bank format | Real format is bank-specific; structure is demonstrated |
| 5 | Notifications are in-app only | Email integration is a production extension |
| 6 | No multi-currency conversion | CurrencyCode stored but conversion rates are out of scope |
| 7 | HalfDay = 0.5 days in balance calculations | Standard HR practice; configurable via LeavePolicy |
| 8 | AttendanceSummary auto-generated on first access | Prevents manual trigger requirement; idempotent operation |
